# Архитектура бэкенда (MVP сразу на микросервисах)

Документ описывает прагматичный вариант: **сразу микросервисная архитектура**, но без тяжёлой инфраструктуры на старте.

Ключевая идея:
- сервисные границы фиксируем сразу;
- запускаемся быстро через Docker Compose;
- сложные компоненты (Kafka/ClickHouse/K8s) подключаем позже, когда появится реальная нагрузка.

---

## 1) Архитектурные принципы

1. **Service per domain**: отдельный сервис на отдельную предметную область.
2. **Database per service**: у каждого сервиса своя БД и своя схема миграций.
3. **API-first контракты**: взаимодействие между сервисами только через публичные API.
4. **Никакой бизнес-логики в Gateway**: только маршрутизация, auth-политики, rate limiting.
5. **Эволюционность**: MVP с минимальным стеком, позже мягко добавляем брокер/аналитику/оркестрацию.

---

## 2) Целевая схема MVP

```text
                   [Web / Mobile Client]
                           |
                           v
                  +---------------------+
                  |     API Gateway     |
                  | (YARP, auth checks, |
                  |  rate limit, routes)|
                  +----+----+----+------+
                       |    |    |
        +--------------+    |    +----------------+
        |                   |                     |
        v                   v                     v
+---------------+   +---------------+    +----------------+
| Identity      |   | Catalog       |    | Booking        |
| Service       |   | Service       |    | Service        |
+-------+-------+   +-------+-------+    +--------+-------+
        |                   |                     |
        v                   v                     v
 [identity-db]       [catalog-db]          [booking-db]

 Опционально в MVP (если успеваем):
   - Chat Service (+ chat-db, SignalR)
   - Media Service (+ S3-compatible storage)
```

## 3) Сервисы MVP: подробно

### 3.1 API Gateway

**Назначение**
- Единая точка входа для фронтенда.
- Роутинг запросов в внутренние сервисы.
- Валидация токена и применение policy (role-based access).
- Базовый rate limiting.

**Стек**
- .NET 9, ASP.NET Core
- YARP
- Serilog + OpenTelemetry

**Хранение данных**
- Нет собственной БД.

**Ключевые endpoints (внешние)**
- `/api/auth/*` -> Identity Service
- `/api/catalog/*` -> Catalog Service
- `/api/bookings/*` -> Booking Service
- `/api/chat/*` -> Chat Service (если подключен)

---

### 3.2 Identity Service

**Назначение**
- Регистрация, логин, refresh token.
- Роли: `Client`, `Manager`, `Admin`.
- Профиль пользователя (базовые данные).

**Стек**
- .NET 9, ASP.NET Core Web API
- PostgreSQL + EF Core 9
- OpenIddict (рекомендуется для MVP) или Duende (если лицензия ОК)
- FluentValidation
- BCrypt/ASP.NET Identity password hasher

**Хранение**
- `identity-db` (пользователи, роли, refresh tokens, audit log входов).

**Ключевые API**
- `POST /identity/register`
- `POST /identity/login`
- `POST /identity/refresh`
- `POST /identity/logout`
- `GET /identity/me`
- `PATCH /identity/me`

**Замечания**
- Только этот сервис знает пароли.
- Остальные сервисы доверяют claims из JWT.

---

### 3.3 Catalog Service

**Назначение**
- Каталог туров/отелей/направлений.
- Фильтрация, сортировка, пагинация.
- Карточка тура и календарь цен.

**Стек**
- .NET 9, ASP.NET Core Web API
- PostgreSQL + EF Core 9
- Redis (опционально в MVP для кэша популярных запросов)
- FluentValidation

**Хранение**
- `catalog-db` (туры, отели, направления, цены, availability, медиа-ссылки).

**Ключевые API**
- `GET /catalog/tours`
- `GET /catalog/tours/{id}`
- `GET /catalog/directions`
- `POST /catalog/tours` (manager/admin)
- `PUT /catalog/tours/{id}` (manager/admin)
- `PATCH /catalog/tours/{id}/prices` (manager/admin)

**Поиск в MVP**
- Сначала SQL + индексы (trigram/fulltext where needed).
- Elasticsearch переносится в V2 при росте каталога и сложных фильтров.

---

### 3.4 Booking Service (CRM-ядро MVP)

**Назначение**
- Создание заявок клиентом.
- Обработка заявки менеджером.
- Избранное.
- Статусы заявки: `New -> InProgress -> ProposalSent -> Confirmed -> Closed/Cancelled`.

**Стек**
- .NET 9, ASP.NET Core Web API
- PostgreSQL + EF Core 9
- FluentValidation
- BackgroundService для outbox-публикации доменных событий

**Хранение**
- `booking-db`:
  - заявки;
  - история статусов;
  - предложения туров;
  - избранное;
  - outbox таблица событий.

**Ключевые API**
- `POST /bookings`
- `GET /bookings/my`
- `GET /bookings/{id}`
- `PATCH /bookings/{id}/status` (manager)
- `POST /bookings/{id}/proposal` (manager)
- `POST /favorites/{tourId}`
- `DELETE /favorites/{tourId}`

**События (MVP)**
- `BookingCreated`
- `BookingStatusChanged`
- `ProposalSent`
- Пишутся в outbox и обрабатываются воркером.

---

### 3.5 Chat Service (опционально для MVP, желательно для travel domain)

**Назначение**
- Чат клиент <-> менеджер в рамках заявки.
- История сообщений.
- Системные сообщения (например, "изменился статус заявки").

**Стек**
- .NET 9, ASP.NET Core + SignalR
- PostgreSQL (история)
- Redis backplane (только если >1 инстанса сервиса)

**Ключевые API/Hub**
- `/hubs/chat`
- `GET /chat/booking/{bookingId}/messages`

**Примечание**
- Если сроки жмут: MVP V1 без отдельного Chat Service, сообщения как комментарии в Booking.

---

### 3.6 Media Service (опционально в MVP)

**Назначение**
- Загрузка и выдача файлов/фото.
- Хранение в S3-compatible storage.

**Стек**
- .NET 9, ASP.NET Core
- MinIO (или S3)
- ImageSharp (опционально thumbnails)

**Ключевые API**
- `POST /media/upload`
- `GET /media/{id}`
- `POST /media/presign`

**Безопасность**
- Ограничение MIME/размера.
- Presigned URL с TTL.

---

## 4) Что исключаем из MVP (осознанно)

Исключаем на старте, чтобы не тормозить запуск:
- Kafka
- ClickHouse
- Kubernetes
- сложные multi-region и advanced autoscaling сценарии

Это не "отказ навсегда", а перенос на этап роста.

---

## 5) Межсервисное взаимодействие

Для MVP:
1. **Синхронно**: gRPC (внутренние вызовы между сервисами).
2. **Асинхронно**: outbox + фоновый publisher (без обязательного брокера в V1).

Далее (V2):
- подключаем RabbitMQ + MassTransit;
- переводим критичные интеграции на события;
- расширяем gRPC-контракты на новые hot-path сценарии.

---

## 6) Инфраструктура MVP

```text
Docker Compose
├── gateway
├── identity-service        -> postgres (identity-db)
├── catalog-service         -> postgres (catalog-db)
├── booking-service         -> postgres (booking-db)
├── chat-service (optional) -> postgres (chat-db)
├── media-service (optional)-> minio
├── redis (optional)
└── postgres (можно 1 инстанс, разные БД на MVP)
```

**Важно:** даже при одном physical Postgres инстансе — разные логические БД/схемы по сервисам.

---

## 7) Безопасность и доступ

- JWT access + refresh.
- Role-based авторизация: `Client`, `Manager`, `Admin`.
- CORS только для доверенных origin.
- Rate limiting в gateway.
- Маскирование персональных данных в логах.
- Audit trail для критичных действий менеджера (смена статуса, отправка предложения).

---

## 8) Наблюдаемость (минимум для MVP)

- **Логи**: Serilog (structured logs, correlation id).
- **Трейсы**: OpenTelemetry traces между gateway и сервисами.
- **Health checks**: `/health/live`, `/health/ready` у каждого сервиса.

Prometheus/Grafana/Jaeger можно добавить на втором этапе.

---

## 9) Рекомендованная внутренняя структура сервиса

```text
ServiceName/
├── Domain/
├── Application/
├── Infrastructure/
└── API/
```

Паттерны:
- Clean Architecture
- CQRS (только где реально полезно, без фанатизма)
- Repository + Unit of Work
- Centralized error handling (ProblemDetails)

---

## 10) План эволюции без смены архитектуры

### Этап V1 (MVP, 6-10 недель)
- Gateway + Identity + Catalog + Booking
- Chat/Media по приоритету
- Compose, без K8s

### Этап V2 (после первых пользователей)
- RabbitMQ + MassTransit
- Notification Service как отдельный сервис
- Elasticsearch при росте каталога

### Этап V3 (масштабирование)
- Analytics Service (ClickHouse/Timescale)
- Kubernetes
- расширенная observability и autoscaling

---

## 11) Критерий успеха MVP

Пользователь должен уметь:
1. зарегистрироваться/войти;
2. найти тур в каталоге с фильтрами;
3. создать заявку;
4. получать ответы менеджера (через чат или комментарии);
5. видеть статус заявки до подтверждения.

Если это работает стабильно, архитектура выбрана правильно.
